## Compact rules

### openspec-git-discipline

- OpenSpec state a downstream phase depends on must be on `main` before that phase; apply is allowed from branch/worktree only if the same proposal change is already on `main`.
- Archive only from `main` after implementation is merged back.
- Before apply: run `git status --short`; ensure no uncommitted proposal files for the target change under `openspec/changes/`.
- Before the next `continue` artifact: prompt to commit completed work or explicit consent to proceed uncommitted.
- Do not create commits, branches, or merges unless the user explicitly asked.
- Worktree or local branch visibility does not prove the proposal reached `main`.

### architectural-decision-records

- One significant decision per ADR; split bundled decisions.
- Honor `preferences.md` `preferred-style`; if unset, ask once, then persist the choice.
- Pick templates from `templates/` for the chosen style.
- Document known facts only; use `Unknown` or short questions instead of inventing context.
- Supersede with explicit links; keep accepted history readable.

### c4-diagrams

- Confirm purpose (existing vs new), diagram format (ASCII vs Mermaid), and rigor (strict C4 vs lightweight) when not already fixed.
- If the user skips questions, record assumptions and keep the first diagram small.
- For existing systems, read code before drawing boundaries.
- Prefer valuable abstraction levels over exhaustive decoration.

### gherkin-authoring

- Detect whether the scope is a whole `.feature`, a fenced `gherkin` block, or embedded prose; preserve non-Gherkin wrappers unless asked to edit them.
- Use concrete scenarios (often 3–5 steps) and observable outcomes; domain wording in steps, not implementation trivia.
- Prefer the smallest structure that fits: Feature, optional Rule/Background, Scenario or Scenario Outline + Examples.
- Validate syntax, duplicated steps, and table/doc string layout before returning.
